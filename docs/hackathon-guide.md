### 해커톤 심사 기준

#### 심사 요건 (필수)

심사 대상에 포함되기 위한 필수 조건입니다.

| 요건 | 설명 |
|------|------|
| **OpenAI API / SDK 사용** | 프로덕트의 핵심 기능에 OpenAI 기술을 활용 |
| **실행 가능한 데모** | 아이디어 수준이 아닌, 동작하는 데모(영상 또는 라이브)를 README.md 등에 명시 |

#### 중점 평가 요소

심사 배점에 중요하게 반영됩니다. 필수 구현은 아니지만, 완성도 있게 구현할수록 높은 점수를 받습니다.

1. **멀티에이전트 협업 구조**
   - 단순 호출을 넘어 에이전트들이 유기적으로 협업하며 복잡한 문제를 해결하는 구조인가?

2. **기업용 (B2B / 사내용) 시나리오**
   - 실무 현장의 구체적인 Pain Point를 해결하며, 도입 시 효용(비용 절감, 생산성 등)이 명확한가?

#### 세부 심사 기준

##### 1. 아이디어 - 30%

> "진짜 문제를 AI가 가장 잘할 수 있는 방식으로 풀고 있습니까?"

- **시장성 & ROI:** 가상의 상황이 아닌, **실제 B2B/사내 환경**에서 비용을 지불할 만큼의 가치(Pain Point 해결)가 있는가?
- **AI 필수성:** 룰베이스(Rule-based)가 아닌, 생성형 AI와 에이전트가 반드시 필요한 문제인가?

##### 2. 기술 구현 - 40%

> "복잡한 기술을 자랑하는 것이 아니라, 문제를 해결하기 위해 최적화했습니까?"

- **멀티에이전트**
  - 에이전트 간의 **협업 워크플로우**가 안정적인가?
  - 에러 발생 시 스스로 문제를 해결하거나 작업을 재분배하는가?
- **도구 사용**
  - Function Calling 시 스키마 검증과 예외 처리가 견고한가?
- **최적화 & RAG (선택)**
  - 데이터 특성에 맞는 검색 전략(청킹, 인덱싱)을 사용했는가?
  - 답변의 출처를 명시하여 할루시네이션을 방지하는가?

##### 3. 완성도 - 20%

> "실무에서 바로 사용할 수 있을 만큼 안정적입니까?"

- **안정성 (Reliability):** 예상치 못한 입력(Edge Case)에도 봇이 중단되지 않고 대응하는가?
- **실용주의 (Pragmatism):** B2B 현장에서 수용 가능한 응답 속도(Latency)와 비용 효율성을 고려했는가? (오버엔지니어링 지양)

##### 4. 문서화 - 10%

- 제3자가 이해하고 실행할 수 있도록 README 및 아키텍처 문서가 명확한가?

#### 가산점 요소

| 항목 | 설명 |
|------|------|
| **Safety** | 가드레일 적용 (보안, 비윤리적 발언 차단) |
| **Advanced UX** | 단순 텍스트를 넘어선 UI 상호작용 (Human-in-the-loop 등) |
| **Observability** | 로그 추적 및 디버깅 환경 구축 |
